## [0.8.0] - unreleased (in progress)
0.8.0 is the defensible / unattended-grade release. A third red-team showed the agent and the things that guard it shared one writable trust domain; the fix is structural, not another string check. v0.8.0 splits that domain into rings (Ring 0 in-process tripwire + deterministic gates, Ring 1 sandbox + read-only control plane, Ring 2 repo-scoped secret-free credential, Ring 3 forge required-review) and adopts one rule: the plugin provisions each ring it can, or it refuses to run unattended without it. The unattended / auto-promote path refuses by default below the hardened tier; an attended local run can proceed at a lower tier only via an explicit `--i-accept-reduced-assurance` escape hatch (logged, labeled REDUCED-INDEPENDENCE). POSIX hardening ships first; on Windows the preflight refuses the local-hardening tiers and points at the devcontainer / WSL or the OS-independent forge guarantee. GitHub is first-class; GitLab parity is a documented section.
### Added
- A preflight prober + a unified trust-tier model (`hooks/preflight.mjs`, wired as a `SessionStart` hook that hard-blocks on a non-zero exit). On every loop start it measures the real assurance of the setup: write-probes every `protectedPaths` entry and the plugin `hooks/` dir as the agent uid; checks `prodBranch` for a no-bypass ruleset; probes reviewer liveness and principal. It classifies a tier (T0 ATTESTED -> T1 LIVENESS -> T2 SEPARATED -> T3 HARDENED), feeds the *detected* tier into the fail-closed promotion floor, and refuses unattended auto-promotion below the configured floor (T3 by default), escalating to `FOR-REVIEW.md` / `turn: human`.
- A presence-verify liveness + principal roster (`hooks/presence-verify.mjs`) that replaces the file-only `readLeases`. A reviewer is "present" iff a process is provably alive (a held `flock` the kernel releases on death, or an open unix socket / named pipe) AND owned by a different principal (POSIX uid via `fs.stat().uid`; Windows owner-SID / pipe-peer-SID is the documented fast-follow, since `fs.stat().uid` is always 0 on Windows). This closes the forgeable-lease scenario (P0-3): killing the reviewer but leaving a fresh `reviewer.lease.json` now reads as ABSENT and blocks the merge, where v0.7.x passed it.
- `harden-control-plane` at init (`hooks/harden-control-plane.mjs`, wired into `autonomy-init.md`): materialize-then-lock every protected path (`chattr +i` plus non-owner-uid guidance on POSIX; ACL / owner on Windows). Materializing first is the point: you cannot read-only-protect a file that does not exist yet, which is how a missing `settings.json` got an injected hook.
- No-bypass prod ruleset provisioning, generalized in `autonomy-init.md` from "warn and continue" to "provision or refuse." It provisions a branch ruleset on `prodBranch` with `pull_request` + `non_fast_forward` + `deletion`, `enforcement: active`, and an empty `bypass_actors` list so the rule binds even admins and the agent's repo-write token is refused server-side. `gate.requireProdProtection` now defaults true, so the preflight refuses unattended when the rail is absent, with `--i-accept-reduced-assurance` as the single attended escape hatch (the same flag for every reduced-assurance tier; there is no second flag).
- A gate-guard HEAD-runner (`hooks/gate-guard-runner.mjs`) that resolves `git push origin HEAD` to the concrete branch via git before the pure `decide()`, keeping `decide.mjs` pure. Defense-in-depth; the ruleset is the barrier, and the runner documents the `$VAR` limit it cannot close.
- A sandbox tier: a one-command `launch-sandboxed` wrapper over `@anthropic-ai/sandbox-runtime` (allow-only writes scoped to the workspace minus the control plane, plus an egress allowlist that also bounds the researcher web-fetch), with the sandbox config itself outside the writable set so the agent cannot self-disable it, AND a plugin `.devcontainer/` with a default-deny egress firewall as the opt-in tier (gVisor/Firecracker microVM documented for untrusted work).
### Changed
- `gate.requireProdProtection` defaults to `true`. The unattended / auto-promote path refuses below the hardened tier by default. This is the product change from v0.7.x: init's "warn and continue" on a missing prod rail becomes "refuse unattended," and the only relaxation is the explicit, logged, attended escape hatch.
- UPGRADE NOTE: after `/autonomy-upgrade`, an unattended loop on a repo without a no-bypass prod ruleset, a read-only control plane, and a separate live reviewer is refused at the preflight rather than run at a tier it cannot back. Run `/autonomy-init` to provision the rings, or pass `--i-accept-reduced-assurance` for an attended local run.
### Security
- These changes are driven by a third independent red-team of the running repo. Its findings shared one root cause (a single writable trust domain) and are addressed by provision-or-refuse rather than by patching strings: P0-1 (edit away the gate / sandbox-and-`cd` bypass) is handled by the write-probe refuse plus the OS-level sandbox and the read-only control plane; P0-2 (`$VAR` and `HEAD` push to prod) is handled at the server by the no-bypass ruleset, with the HEAD-runner as defense-in-depth and the `$VAR` residual named honestly; P0-3 (forgeable reviewer lease) is handled by the liveness + principal prober feeding the fail-closed floor. P1-4 (a regression riding into the weaker mutation gate) was fixed in code last cycle: `classify-bite` now routes any modified-source commit to golden-revert, and the exact PoC diff is a permanent regression test. The honest residuals are stated plainly in SECURITY.md: same-uid local setups are only raised by signing/liveness (the forge is the only unattended-grade independence), the `$VAR` push is closed only by the server ruleset, a broad egress domain remains an exfiltration channel, and the regex is a tripwire, never a boundary.

## [0.7.0] - unreleased (in progress)
0.7.0 supersedes 0.6.0's config-flag role ladder and LLM-judged plan gate with composable auto-detected roles (Release A), a deterministic floor under every LLM judgment (Release B), and three launch features: a convergence terminator, a greenfield mutation-bite, and single-CLI mode. This section accumulates work as it lands; the version is cut and tagged once the launch features are wired and live. See `autonomy-loop-v0.7.0-PLAN.md` for the risk-ordered rollout (nothing that can silently pass bad code ships without a flag defaulting to current behavior, a shadow period, a corpus check, and a one-switch rollback).
### Changed (defaults: rigor ON)
- Per the defaults philosophy, the v0.7.0 rigor gates ship ON, not off: a gate hidden behind an off-by-default flag is, for almost every user, a gate that does not exist, and claiming a toggled-off feature publicly is dishonest. `gate.selfMutate` is ON; the oracle-strength gates (`atsg` acceptance-test mutation-kill + `assert-classify` weak-assertion) BIND on every reviewed wave; the money-path floor (`mpid`) and the verify-gate router (`gate.verifyGate: "govern"`) govern by default. The speed opt-out (`speed.optOut`) is the only relaxation and it lightens ONLY plain additive, reversible waves, never money-path or irreversible ones.
- UPGRADE NOTE for existing 0.6.x users: after `/autonomy-upgrade` these gates are active, so a wave that ships a test which runs a line but pins nothing about it now correctly bounces where 0.6 would have passed it. The verify-gate router is fail-closed (it never returns a pass without a recorded killed-mutant or a clean RED), so it can only block, never silently pass; set `gate.verifyGate` to `shadow` to watch it before it governs, or `off` for the pre-0.7 golden-revert-only behavior. The greenfield path needs `gate.coverage` set to score covered lines.
### Added
- Four launch-feature decision cores landed as PROVEN pure cores (fail-closed, no deps, 19 unit tests), wiring staged per the rollout plan: `convergence-core` (an oscillation detector keyed on the gate-failure signature plus a per-task attempt budget and an escalation ladder rescope -> stronger-model -> park, the deterministic "third vote" that closes the churning-loop gap the no-progress breaker misses because the tree-SHA changes every wave); `scope-core` (a planner-set scope ceiling that forces a mandatory handoff before the builder goes too deep, fail-closed when a ceiling cannot be measured); `classify-bite` (a deterministic router that picks golden-revert vs mutation-bite per commit and surfaces which one ran, with UNCLASSIFIABLE -> exit 2); `mutation-bite` (the greenfield mutation-kill gate for brand-new code the revert cannot handle, holding the global invariant that no input yields exit 0 without a recorded killed mutant, fuzz-verified).
- The 14 Release A (composable auto-detected roles: role-registry, presence, turn-scheduler, safety-floor, migrate-roles, concurrency-harness, gate-guard-promotion, ux-banner) and Release B (the ground-truth floor: mpid money-path detector, assert-classify weak-assertion detector, atsg acceptance-test mutation-kill, quarantine, oracle-separation) decision cores are now landed in `hooks/` with their tests (126 green), plus a `coordination-mode` core for single-CLI subagent mode (default off, REDUCED-INDEPENDENCE trust tier, refuses to run if the deterministic gate is off so nothing independent is lost). These are pure cores; the repo-touching WIRING (presence I/O, the adaptive turn scheduler, the safety-floor promotion block, the reviewer oracle-strength gates, the researcher quarantine sandbox, single-CLI subagent orchestration, and the bite router via `gate.verifyGate=off|shadow|govern`) is staged behind flags per the rollout plan, with scratch-repo acceptance tests before anything governs.
- The repo-touching GLUE runners are now built and scratch-repo verified (all default-safe, not yet invoked by the prompts): presence/roster lease I/O plus an adaptive turn-scheduler baton with epoch fencing (`presence-io.mjs`, `baton-io.mjs`, 8/8 ACs); a safety-floor promotion block that blocks a merge or a push to prod without a live independent reviewer, brute-forced across every roster (`promotion-guard.mjs`, 22 tests); a researcher quarantine wrapper plus a 55-payload adaptive red-team corpus that zero payloads pass (`researcher-quarantine-io.mjs`, `redteam/`); the bite router plus the greenfield mutation-bite RUNNER in shadow mode (`verify-gate.mjs` + `mutation-bite.mjs` main(), every KG/KB fixture green; `gate.verifyGate` GOVERNS by default and is fail-closed, with `shadow` as a watch mode and `off` as the pre-0.7 behavior); and the single-CLI subagent templates with a read-only reviewer guard (`templates/agents/`, `templates/hooks/reviewer-readonly-guard.sh`). The new config knobs (coordination, safety, subagent, convergence, scope, speed, gate.verifyGate) are documented in `autonomy.config.example.json`; per the defaults philosophy the rigor gates ship ON (see Changed above), with `speed.optOut` the only, additive-only relaxation.
### Fixed
- Security hardening of the enforcement hooks from a second adversarial red-team, each fix paired with an adversarial test. The gate-guard now blocks a prod-branch push from ANY remote (not just `origin`, so `git push upstream main` is caught), remote-branch deletion (`--delete` / `-d`), local ref destruction (`branch -D`, `update-ref -d`, `reflog expire`, `checkout --orphan`), and bundled force flags (`-uf`); its protected-path denylist is broadened (`dd`, `install`, `ln`, numbered-fd redirects) and matched against a quote-stripped copy so a split path (`test/gol''den/x`) cannot evade it, while still NOT over-blocking safe pushes (`main:feature`, a `maintenance` branch, commit messages that mention main). The bite's `classifyOutcome` no longer reads a crash log that merely contains "assert" / "expected" / "failing tests" as a caught assertion: those bare markers were dropped and runtime-crash markers added, so a reverted fix that crashes fails CLOSED to cannot-verify instead of faking a caught bite. `mutate.mjs` and `patch-coverage.mjs` validate `--base` against the bite's `^[0-9A-Za-z_./^~-]+$` ref guard before interpolating it into `git diff`, and `parseDiff` treats `+++ ` as a file header only when it follows `--- ` so added content beginning with `++` is no longer dropped. The gate-guard remains a best-effort denylist tripwire by design; the `SECURITY.md` threat model still applies.

## [0.6.0] - 2026-06-18
### Added
- The plan lane: a Planner terminal (T3) that turns research into gated, buildable specs and feeds the self-feeding build loop. This is the "feeder in front of the self." When the backlog drains, the loop no longer leans only on the Builder's ad-hoc MODE A ideation; a dedicated Planner researches wide (fan-out across lenses: product gaps, competitors, marketing, SEO, pricing, UX), then GRILLS one spec into a detailed doc with clear, falsifiable ACCEPTANCE CRITERIA and a goal-ready build prompt, and hands it to the Reviewer to screen before the Builder may touch it. `commands/planner.md` + `templates/SPEC.md`. Opt-in via `roles.planner`, off by default.
- The plan gate. The Reviewer (T2) now screens every spec on a five-check rubric (sourced, has a falsifiable acceptance test, scoped, risk tier, real ROI) before the Builder builds it. Additive specs pass autonomously; anything touching the money path, the frozen invariant, new infra/secrets, or anything irreversible PARKS to `FOR-REVIEW.md` for the owner's GO. This closes the one hole a planning layer would otherwise open: a plan that bypasses the gate. The rubric is anchored, not free-form, because the research on agentic self-correction is blunt that judgment without a ground-truth signal can degrade output, and the mandatory acceptance test is that signal.
- Three shapes, one dial. The `roles` config block: both false is the classic 2-terminal self-feeding loop, byte-for-byte v0.5 (the Builder's MODE A is the idle feed). `roles.planner` true is the recommended 3-terminal shape (Planner + Builder + Reviewer on the single `LOOP-STATE.md` baton; MODE A becomes the fallback floor so the loop never starves). `roles.research` true as well is the 4-terminal power mode: a dedicated Researcher (T3, `commands/researcher.md`) fills the idea pool on a parallel upstream baton (`templates/PLAN-STATE.md`) while the Planner (T4) drains it, two decoupled lanes joined by an approved-spec queue under a strict single-writer-per-file discipline.
- Growth routing. Non-code research output (competitor battlecards, positioning briefs, SEO/content plans, pricing recommendations) is drafted to `GROWTH.md` and parked to `FOR-REVIEW.md` for the owner to publish, since public content and pricing are gate-list and irreversible. Marketing still gates, it just gates to the owner instead of to the build.
- New model knobs `models.researcher` (cheap, for the fan-out) and `models.planner` (Opus-class, the spec is the product); a plan-lane circuit breaker `breaker.maxPlanEpochs` / `breaker.maxPlanNoProgress` (the feeder's runaway guard, independent of the build-lane breaker); and `templates/RESEARCH-LANE.md` tightened into the research-phase contract (the lenses, claim-by-claim sourcing with URL verification, dedup plus TTL on the idea pool, diverge-then-converge scoring).
### Changed
- `commands/builder.md`: feed precedence. What to build next is (1) `pending-for-builder`, then (2) the Planner's approved spec when `roles.planner` is on, with MODE A demoted to a fallback floor used only in classic mode or when the Planner has stalled, so `tasks/IDEAS.md` always has a single writer.

- `templates/LOOP-STATE.md`: the baton now documents the `planner` turn and a `pending-for-screen` slot, both unused when `roles.planner` is false.
- `hooks/migrate-config.mjs`: idempotently tops up `roles` (off), the two `breaker` plan fields, and `models.researcher`/`models.planner` (only when a models block exists) for existing installs, never overwriting a set value and never resetting a baton. Six new migrator tests.
### Fixed
- Plan-lane wiring, caught by an adversarial red-team of the repo before public use: the 3-terminal cycle now closes (on a code-review PASS the Reviewer hands the baton back to the Planner when `roles.planner` is on, instead of wedging in builder<->reviewer and starving the feeder); `/autonomy-init` scaffolds `GROWTH.md` and `tasks/IDEAS.md`; `pending-for-screen` defaults empty so a classic 2-terminal install never trips the plan-screen on a placeholder; SECURITY.md gains a real threat model and the README/headline acknowledge the planner.

## [0.5.2] - 2026-06-18
### Fixed
- The mechanized bite was broken on Windows, caught by a live dogfood loop running it for real. `bite.mjs` shells out through `execSync`, which on Windows runs `cmd.exe`, where `^` is the escape character. The internal `git rev-parse <sha>^` and `git diff <sha>^ <sha>` lost the caret, so `<sha>^` resolved to the commit itself, the diff came back empty, and every bite falsely reported "the wave changed no source files" (cannot-verify, exit 2). The runner now uses `<sha>~1` for the parent (tilde is not special to cmd.exe) and detects merge commits caret-free via `git rev-list --parents`, so the bite runs on Windows, macOS, and Linux. Behavior is identical on POSIX (`~1` and `^` both mean the first parent); only the Windows breakage is removed.

Either `/autonomy-upgrade` or a re-run of `/autonomy-init` migrates safely.

## [0.5.0] - 2026-06-17
### Added
- The bite is now a real gate, not just a reviewer instruction. `hooks/bite.mjs` mechanizes it: given a wave's fix and the new test, it reverts only the source change in a throwaway detached git worktree (keeping the test in place), reruns the test, and requires an assertion failure to pass. A test that still passes when the fix is reverted catches nothing and fails the gate. Validated against mutation-testing and regression-test-selection prior art: an exit code is not treated as a valid RED (an assertion failure is distinguished from a build/collect error or a timeout, so a fix that no longer compiles cannot masquerade as a caught bug), a flake guard requires the same failure N times in a row (default 3), the test must be green on the fixed code first, and a test that does not execute the reverted code is an honest cannot-verify rather than a pass. Fail-closed exit codes: 0 pass, 1 stayed-green, 2 cannot-verify. Pure `decideBite` plus `classifyOutcome` core with 17 unit tests; wired into the reviewer gate. No external deps.
- Control-plane hardening: the loop's own `autonomy.config.json`, `.autonomy-coverage.json`, and `autonomy-loop/hooks/` are now in the default `protectedPaths`, so a wave cannot edit away or delete its own gates. (`.autonomy-coverage.json` is safe to protect: the ratchet writes it through Node fs, which the gate-guard does not intercept, only Edit/Write tools and shell redirects.)
- Cross-wave circuit breaker: the reviewer now tracks `epoch`, `no-progress-epochs` (consecutive waves whose HEAD tree-SHA does not change), and an optional `maxBudgetUsd`. On a hard epoch cap, a no-progress run, or a budget overrun it parks the loop to `FOR-REVIEW.md` with `turn: human` instead of looping forever. Tunable via a `breaker` block in the config, and carved out in both prompts as the one sanctioned exception to the no-stand-down rule.
- `autonomy-init` now verifies server-side branch protection on `prodBranch` (a ruleset-aware `gh api rules/branches` check) and offers to set a minimal ruleset (require a pull request, block force-push, block deletion). It warns and continues if `gh` is missing, unauthenticated, the remote is not GitHub, or the user lacks admin, so setup never hard-blocks.
- Docs: a README "Operate (first run)" section (two terminals, the reviewer worktree, start-green, approve-first, how to stop, cost) and an accurate coverage-prerequisite note (jest/vitest provider or Node 22+, otherwise the coverage gates stay off and the loop still runs).
- Self-mutation self-check (builder b1, opt-in via `gate.selfMutate`): before handoff the builder mutates only the lines its diff changed and requires the tests to kill each mutant; a surviving mutant means a test ran the changed line but asserted nothing about it. Text-level, language-agnostic operators (relational boundary, equality, logical, boolean, arithmetic, integer off-by-one) with string and comment contents masked first so a mutation can never land inside a literal; one mutant per line (a line's mutants almost always share a fate); a timeout counts as killed; unviable mutants are excluded from the denominator; survivors are advisory by default because the equivalent-mutant problem is undecidable, with an allowlist for confirmed equivalents and `--block` for a hard gate. Pure `maskCode` plus `mutantsForLine` plus `decideMutation` core with 15 unit tests; reuses the bite's mutate-then-restore model and the patch-coverage diff parser. The builder's upstream complement to the reviewer's bite. Also adds a builder convergence bound: a wave whose gate stays red after 3 fix attempts parks to `FOR-REVIEW.md` instead of looping.

### Fixed
- `coverage-ratchet` now creates the baseline file's parent directory before writing it, so seeding a baseline into a nested path (for example a `.alvg/` subfolder) no longer throws `ENOENT`. A no-op when the baseline sits at the repo root, as in the default config.
- Bite hardening after an adversarial self-review: `classifyOutcome` now fails CLOSED on a genuinely ambiguous non-zero exit (no timeout, build-error, or assertion signature, and no `--assert-regex`) and returns `error` (cannot-verify) instead of assuming a caught assertion, so a silent crash can never masquerade as a real RED. The runner also refuses a merge commit as the fix (diffing one parent would revert the wrong slice), validates the `--fix` ref against shell-metacharacter injection, and only `git rm`s a reverted file that genuinely did not exist at the parent (a failed restore of a pre-existing file now surfaces as cannot-verify instead of silently deleting it).
- Self-mutation (`mutate`) restore is now signal-safe: a `SIGINT`/`SIGTERM` while a mutant is live on disk restores the file before exiting, and a failed restore logs a loud `MANUAL RESTORE NEEDED`, so an interrupted run can never leave the builder's working tree mutated.

## [0.3.2] - 2026-06-16
### Fixed
- Coverage ratchet now gates branch coverage, not just line coverage. Branch percentage was tracked and ratcheted but never checked against a floor, so branch coverage could rot under a flat line number (a new conditional whose else-arm is never tested would slip through while lines held). `decideRatchet` now applies a branch floor symmetric with lines: a drop past epsilon is a regression, an in-band dip holds without rewriting the baseline, lines and branches ratchet up independently, and the branch floor only moves up. A real branch floor with no branch measurement this run is an honest cannot-verify error rather than a silent pass, and a corrupt baseline `branches` value errors instead of disabling the gate. Seven branch-floor tests added (`test/branch-floor.test.mjs`); the existing case that encoded the old un-gated behavior was updated to a within-band dip. 24 ratchet tests total, all green.

## [0.3.0] - 2026-06-16
### Added
- Patch coverage: a FOURTH gate, opt-in via `gate.patchTarget` greater than 0. It scores only the lines a wave changed, using the Istanbul `coverage-final.json`, so new untested code can no longer hide behind a flat or rising global percent (the ratchet's blind spot). Set the bar to 80 for standard or 100 to require every changed line tested so a single bare line cannot ride through. Self-contained Node (`hooks/patch-coverage.mjs`: pure `decidePatch` plus a diff parser plus an Istanbul reader), no external deps, 14 unit tests. Wired into the builder and reviewer gate. ## [0.2.1] - 2026-06-16

## [0.2.0] - 2026-06-16
### Added
- Coverage ratchet: an opt-in THIRD gate (the drift guard), enabled by setting `gate.coverage` to a command that emits an Istanbul `coverage-summary.json` (for example `c8 --reporter=json-summary --reporter=text npm test`). Total line coverage can never fall below a stored baseline in `.autonomy-coverage.json`, and the baseline only ever ratchets up, so coverage holes cannot quietly accumulate wave after wave. It pairs with the per-fix bite: coverage measures execution, not assertions, so the ratchet is the drift layer, never a quality claim on its own.
